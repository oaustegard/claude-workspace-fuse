# Boot Payload Ledger — cost vs. fire rate (issue #84)

Generated by `muninn_utils.boot_ledger` against live Turso + the memory corpus
on 2026-07-10. Regenerate with:

```bash
PYTHONPATH=<repo>/remembering:<repo> python3 -m muninn_utils.boot_ledger
python3 -m muninn_utils.boot_ledger --json   # machine-readable
```

## What this measures

Two numbers per boot-loaded config entry:

- **cost** — exact. Character length from the config table plus a token
  estimate (tiktoken cl100k when importable, else ~4-chars/token). This is what
  the entry spends on *every* boot whether or not it is ever used.
- **fire rate** — a **proxy**. There is no historical log of `config_get` calls
  (that gap is the point of this issue), and the archived session transcripts
  are three stale sessions from one April day — not a usable corpus. The one
  longitudinal record that exists is the memory corpus (~2.2k active memories,
  Dec–Jul). So "fire" is approximated as: how many active memories reference the
  entry's domain (via a curated high-precision term set per key), across how
  many distinct months. It answers *"did this entry's condition actually arise
  in logged work"* — the countable half of the fire-rate question.
- **logged** — exact, but 0 until instrumentation runs. `config_get` now
  increments a `fire_count`/`last_fired` counter for boot-loaded keys when
  `MUNINN_INSTRUMENT_FIRES=1` (see below). After a measurement window the proxy
  can be retired for those keys.

`chars_per_hit` (chars ÷ hits) is the ranking key: high = expensive per unit of
demonstrated relevance = prune queue.

**Kinds** decide how to read the fire column: `trigger` (dispatch line — fire
rate is the whole point; zero → demote), `ops` (semi-passive knowledge — low
fire is a weaker signal), `identity` (profile always-on — fire informational,
not a demotion lever), `catalog` (the voice-signature scan list — curated by
redundancy, not demotion).

## Headline findings

- Boot payload: **51 entries, ~86.7k chars / ~21.7k tokens** — loaded in full on
  every session.
- **No entry is never-referenced.** Every trigger's and ops entry's subject
  arises in the corpus at least a few times. The waste is **not dead triggers**;
  it is **oversized payloads and catalog redundancy.** The two largest entries
  (`muninn-voice-signature`, `voice`) are 23% of the payload by chars.
- The top of the cost/fire table is dominated by **large `ops` payloads**
  (`instruction-provenance`, `operating-imperatives`, `filedrop-implementation`,
  `confabulation-cascade`, `satisfaction-register`) — entries whose *content* is
  reference material, not a compact dispatch trigger. Per the `config_set`
  design note ("if an entry deserves boot-load, write it as a compact trigger …
  Payloads stay reference"), these are the standing demotion **candidates** — a
  compact trigger stays boot-loaded, the payload moves to reference-only and
  loads on demand. This is a judgment call per entry (demoting the payload
  without leaving a dispatch trigger silently loses the capability), so it is
  **recommended, not auto-applied** — see below.
- The aggregate boot total is a **moving target**: during this very run a
  concurrent session grew `satisfaction-register` by ~1.3k chars. An unmanaged,
  actively-written payload with no ledger is precisely the failure #84 names.

## Curation applied (this PR)

**Collapsed the voice-signature "manufactured significance / staged reveal"
cluster.** Eight sibling anti-pattern scans — Heroic-narrator framing,
Drama-line-breaks, Performed significance, Hindsight-knowing contrarian voice,
Negation-first reveal, Suspense-construction reveals, Thesis-shaped/coy section
headers, Pretentious contrarian intellectualism — all encode **one root** ("don't
be coy; state the finding plainly instead of performing the approach to it").
They were each re-deriving that root under a new surface tell. Collapsed into a
single scan, *"Manufactured significance — staging a reveal instead of stating
the finding,"* with every distinct surface tell preserved as a bullet and every
diagnosis date retained.

- `muninn-voice-signature`: **12,755 → 11,880 chars** (−875 chars / ~−219
  tokens), 8 scans → 1 with a preserved tell-list. Applied live via `set_rule`
  (drift_class `narrowing`); audit memory `9b3396dd`, prior value backed up at
  `docs/muninn-voice-signature.pre-84.md`. Revert = restore that file with
  `config_set`.

This is the one attributable, verifiable drop (the aggregate is confounded by
concurrent writes, as noted). All eight scans still fire; no distinct tell lost.

## Go-forward instrument

The real fix for "unmanaged" is to stop guessing fire rate. `config_get` now
records exact fires for boot-loaded keys, opt-in and zero-cost by default:

```bash
MUNINN_INSTRUMENT_FIRES=1   # set for a measurement window (e.g. a week of sessions)
```

It increments `config.fire_count` / stamps `config.last_fired` only for
`boot_load=1` keys (payload reads don't inflate it), best-effort and
self-silencing so it can never break a read. Re-run `boot_ledger` afterward: the
`logged` column carries real counts and supersedes the proxy.
